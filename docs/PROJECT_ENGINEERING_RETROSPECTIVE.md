# SmartCheckIn 工程复盘与学习笔记

> **定位**：以资深前端工程师视角，把本项目从 0 到可用、迭代优化、部署与测试过程中**值得反复遇到的真实问题**整理成一份学习材料。  
> **读者**：希望系统理解「全栈小产品」如何落地、以及移动端 Web 常见坑的同学。

---

## 一、项目是什么（一句话 + 边界）

- **产品**：组织者创建签到活动；参与者通过**地理位置**和/或**动态二维码**在有效时间内签到；有组织、邀请码、记录与统计。
- **技术形态**：单仓 **Vue 3 + Vite** 前端 + **Express + SQLite** 后端；开发期双进程（`concurrently`），生产期 **Node 同机托管 API + `dist` 静态资源**（`NODE_ENV=production` 时由 Express 兜底 SPA 路由）。

更完整的需求与接口约定见：`docs/AI_DEV_PRODUCT_SPEC.md`。

---

## 二、从 0 到「可交付」的典型路径（你在本仓库里能看到的顺序）

1. **规格先行**  
   用产品规格文档锁定：角色、签到方式、错误码、REST 形状、环境变量命名。避免做到一半才发现「组织者/参与者权限」「名单制」等大块需求未对齐。

2. **后端骨架**  
   Express 挂载 `/api/*`，统一 JSON 错误结构（如 `ok: false` + `error.code`）；JWT 鉴权中间件；SQLite 初始化与迁移集中在 `server/db.js`。

3. **前端骨架**  
   Vue Router 划分组织者流 / 参与者流；Pinia 存 token 与用户信息；`fetch` 封装在 `src/api/client.js`，自动带 `Authorization`。

4. **开发联调**  
   Vite `server.proxy` 把 `/api` 转到本机 Node 端口，**避免浏览器 CORS**（见 `vite.config.js`）。注意：**代理读的是环境里的 `API_PORT`**，与 `server/index.js` 监听端口必须一致。

5. **移动端能力**  
   Geolocation、`html5-qrcode` 摄像头、相册识码；这些在**非 HTTPS** 或**壳浏览器**上行为差异极大，需要产品文案 + 技术降级（下文第三节）。

6. **生产构建**  
   `pnpm build` 产出 `dist/`；生产环境 Express `express.static` + `history fallback`，保证刷新子路径仍能回到 `index.html`。

7. **部署与验证**  
   按 `docs/DEPLOY.md`：Nginx 反代、防火墙、PM2、`.env` 中 `JWT_SECRET` 等。无备案 HTTPS 时的临时方案见 `docs/CLOUDFLARE_TUNNEL.md`。

8. **体验与视觉债**  
   `docs/UI_ART_REVIEW.md` 把「设计系统未闭环、a11y、深色页与全局浅色割裂」等问题结构化列出，适合作为下一波前端的 backlog。

## 三、开发过程中「有价值的问题」归类（建议当作面试/复盘题）

### 3.1 同源、HTTPS 与敏感 Web API

| 现象 | 原因 | 本项目中的处理思路 |
|------|------|-------------------|
| 本机 `http://局域网IP:5173` 调定位/摄像头失败或受限 | 许多 API 要求**安全上下文**（HTTPS 或 localhost） | 文档引导用 HTTPS 隧道测试；生产用正规证书或 Cloudflare 隧道 |
| 微信内置浏览器、百度浏览器行为不一致 | 壳浏览器裁剪权限、WebView 策略不同 | 参与者页对百度等给出**明确文案**；扫码提供**拍照识别 / 手动输 token** 等降级路径 |

**学习点**：前端「能跑」不等于「在目标用户浏览器里能跑」；要在 UI 上**承认限制**并给替代方案。

### 3.2 剪贴板 API 与国产浏览器

| 现象 | 原因 | 处理 |
|------|------|------|
| `navigator.clipboard.writeText` 抛错或不可用 | 非安全上下文、内核未实现、或用户未授权 | 实现 **`copyToClipboard` 降级**：先 Clipboard API，失败再用 `textarea` + `document.execCommand('copy')`；仍失败则展示**只读输入框**方便长按全选（`src/utils/copyText.js` + 组织者仪表盘） |

**学习点**：Clipboard 要写**策略矩阵**：安全上下文检测 → 现代 API → 传统 execCommand → 最后 UI 兜底。

### 3.3 二维码内容设计：整段 URL vs 纯 token

| 问题 | 说明 |
|------|------|
| 扫码结果经常是完整 URL | 不同 App 返回格式不统一 |
| 若只支持「纯 token」 | 用户从相册/第三方扫码结果粘贴时会失败 |

**处理**：前后端约定**同一套解析**：从 querystring 抽 `token`，失败再退回整段字符串；前端 `src/utils/qrPayload.js` 与后端 `server/utils/qrPayload.js` 注释中强调保持一致。

**学习点**：二维码载荷要**面向真实扫码生态**设计，并在服务端**再校验/再抽取**一层，防客户端差异。

### 3.4 地理定位：精度、超时与「等多久」

参与者页使用 `watchPosition` 等逻辑时，常见坑包括：

- 首次定位慢、室内精度差；
- 一直等「更好精度」导致用户焦虑；
- 需在超时、精度阈值、**改善停滞**之间做平衡（你当前实现里包含较长上限与提前结束策略，具体数值以 `SessionDetailView.vue` 为准）。

**学习点**：地理签到是 **UX + 阈值调参** 问题，不是「调一次 API」就结束。

### 3.5 开发代理 vs 生产同域

- **开发**：`VITE_API_BASE_URL` 常为空，请求走相对路径 `/api`，由 Vite 代理到 Node。  
- **生产**：`api/client.js` 的 `apiBase()` 会对「只写了源站根 URL」的情况**自动补 `/api`**，减少部署时配错 env 的概率。

**学习点**：封装 baseURL 时要把「人类容易配错的形态」考虑进去（末尾斜杠、是否含 `/api`）。

### 3.6 单体部署与 SPA 路由

生产环境 Express 对非 `/api` 路径 `sendFile(index.html)`，避免用户刷新 `/participant/...` 时 404。

**学习点**：**任何** History 模式的 SPA 部署都要在服务器层做 fallback，否则「深链接」必挂。

### 3.7 安全与运维

- **JWT_SECRET** 不可用仓库默认值；生产用强随机。  
- **数据库文件权限与备份**：SQLite 单文件，备份/迁移策略要心里有数。  
- **防火墙**：只暴露 80/443，Node 端口仅本机或内网。

## 四、资深前端视角的「项目总结」

### 4.1 做得好的地方

- **规格文档可执行**：`AI_DEV_PRODUCT_SPEC.md` 把 MUST/SHOULD、技术栈、错误语义写清，适合人机协作迭代。  
- **真移动场景**：定位 + 扫码 + 壳浏览器文案，说明团队在真实手机上验过。  
- **工程化够用**：单仓脚本、`proxy`、生产静态一体化，部署文档具体到端口与防火墙。  
- **防御性解析**：QR token、API base URL 等有多层兼容。

### 4.2 技术债与风险（诚实清单）

| 区域 | 说明 |
|------|------|
| 设计系统 | 大量页面级 inline style，与全局 token 并存，长期维护成本高（详见 `UI_ART_REVIEW.md`） |
| 测试 | 规格里建议的 Haversine、QR 校验等单元测试若未全覆盖，重构时易回归 |
| 类型 | 全 JS 在快速迭代时爽快，规模再大时可考虑渐进 TS 或 JSDoc 严格模式 |
| README | 根 `README.md` 仍是 Vite 模板，对新同学不友好——可用三句话 + 链到本文与 DEPLOY |

### 4.3 建议的优化路线（按性价比排序）

1. **P0 体验**：继续扫国产浏览器差异（剪贴板、摄像头、定位权限），统一走 `utils` 层 + 统一错误提示组件。  
2. **P0 可维护**：把重复布局从 inline style 抽到 utility class 小组件（不必一次上设计系统大改）。  
3. **P1 质量**：为 `qrPayload`、距离计算、邀请码校验等纯函数补 `node:test` / vitest。  
4. **P1 可观测**：生产请求 id、关键业务日志（签到失败原因聚合），便于线上排障。  
5. **P2 工程**：CI 里 `pnpm build` + 基础 lint；环境分 staging/prod。  
6. **P2 视觉**：按 `UI_ART_REVIEW.md` 处理深色二维码页与全局浅色的一致性、a11y。

## 五、建议你这样「学这个项目」

1. **先读三条线**：规格（产品）→ `server/index.js` + 某一资源路由（数据流）→ `src/router/index.js` + `api/client.js`（前端数据流）。  
2. **再跟一条用户路径**：登录 → 创建活动 → 大屏二维码 → 另一设备参与者签到（地理 + QR 各走一遍）。  
3. **刻意练习两个坑**：① 用 HTTP 访问看定位/摄像头提示；② 用限制剪贴板的环境测复制链接降级。  
4. **最后做一个小改动**：例如给复制成功加 toast、或给某列表加空状态，体会设计 token 与组件抽取的必要性。

# 人脸识别签到 — 产品需求文档（PRD）

**版本**：1.0  
**前置规格**：参见 `AI_DEV_PRODUCT_SPEC.md`（原「非目标」中的人脸识别本条由本 PR 覆盖并实现为可配置能力）  
**状态**：已实现（与本仓库代码同步）

---

## 1. 背景与目标

### 1.1 为什么要加「人脸识别签到」

- **组织者诉求**：在活动场景下区分「本人在场」，降低代签风险；与地理位置、二维码互为补充——地理解决「到人附近」，人脸提供「本人在设备前完成核验」的主观约束。
- **产品目标**：在不引入硬件闸机、不依赖第三方支付级活体SDK的前提下，在 **Web/H5 + 自建 Node/SQLite** 技术栈内提供 **可选、可用、可调参** 的人脸核验能力。
- **非目标**：不承诺金融级活体防攻破；不代替法律意义上的身份核验；不向监管场景默认可用。

### 2. 产品原则（产品经理视角的稳定可靠）

| 原则 | 说明 |
|------|------|
| **诚实披露** | UI 与 PRD 明确：基于浏览器计算的 **Face descriptor（128 维向量）比对**，易受光线、角度与照片翻拍影响；**HTTPS + 前置摄像头权限**是实现前提。 |
| **先录入、后核销** | 参与者须先在站内完成「人脸样本录入」，服务端仅存 **向量 JSON**，不传图存底（降低隐私与带宽风险）。录入与签到分离，避免活动当天首登失败扎堆。 |
| **组织者可控** | 签到方式枚举扩展：`FACE`（仅人脸）、`GEO_FACE`（地理围栏成功 **且** 人脸比对成功，**一次请求原子提交**）；历史 `GEO` / `QR` / `BOTH` 行为保持不变。 |
| **失败可观测** | 失败记录写 `failure_code`（如 `face_mismatch`、`face_not_enrolled`），组织者统计按方式区分 `face`、`geo_face`。 |
| **工程可演进** | 阈值、比对算法留在服务端单侧；未来可替换为云端活体或设备端 ASE，不改变「录入向量 + 比对」的产品形态。 |

### 3. 用户画像与典型场景

- **组织者**：上课/例会，希望到场同学本人签到；可选用「地理+人脸」防窗外代签。
- **参与者**：已登录；活动前在「录入人脸样本」页完成采样；活动时按提示打开摄像头并完成一次取样。

## 4. 方案说明（为什么选择 descriptor 比对）

- **为何不直接上传整张脸图**：SQLite 存二进制大图不利于备份与安全响应；且无 GPU 服务端时不便跑重型模型。
- **为何向量在服务端比对**：防简单的「直接改 HTTP 结果为成功」不可用——至少需在请求中携带与高维向量一致性的 payload（仍非金融安全，强于纯前端放行）。
- **已知局限（需在帮助文案中体现）**：攻击者若获得他人 enrolled 向量（中间人 + 反编译）理论上可重放；MVP 阶段与「校园签到」风险模型匹配，高敏场景需后续接云端活体或专用 App。

## 5. 功能需求

### 5.1 参与者

| ID | 需求 | 验收 |
|----|------|------|
| P-1 | 提供「人脸样本录入」独立入口；成功后在个人维度可再次覆盖录入。 | `POST /api/users/me/face` 保存 128 维数组；首页/账户区可见入口。 |
| P-2 | 活动模式为 `FACE` 时，仅展示人脸签到卡片；未完成录入时阻断并提示去录入。 | 接口返回 `face_not_enrolled`；文案清晰。 |
| P-3 | 活动模式为 `GEO_FACE` 时，先完成与现网一致的定位流程，再做人脸取样，**一次提交** `POST .../checkin/geo-face`。 | 围栏外或人脸不匹配均失败并记原因；成功仅一条成功记录 `method=geo_face`。 |
| P-4 | 模型资源从公共 CDN 加载（可配置）；失败时提示网络与 HTTPS。 | 首次打开摄像头前 `loadFaceModels()`。 |

### 5.2 组织者

| ID | 需求 | 验收 |
|----|------|------|
| O-1 | 新建/编辑活动可选「仅人脸识别」「地理+人脸识别」；`GEO_FACE` 必须配置地理围栏。 | 与 `GEO` 相同校验 `geo_config`。 |
| O-2 | 统计与签到记录中可区分 `face` / `geo_face` 与既有 `geo` / `qr`。 | Stats/Records/Export 行为一致。 |

### 5.3 服务端

| ID | 需求 | 验收 |
|----|------|------|
| S-1 | 用户表增加 `face_descriptor`（JSON 文本，128 个有限浮点）。 | 迁移幂等。 |
| S-2 | 欧氏距离阈值默认 **0.55**（与 face-api 常见 FaceMatcher 量级一致，可抽常量）。 | 单元测试或手工对比两条向量。 |
| S-3 | `POST /checkin/geo` 在 `GEO_FACE` 模式下 **禁止** 单独成功（避免「只签地理」绕过人脸）。 | `mode_not_allowed` 提示使用联合签到。 |

---

## 6. 交互与文案要点

- 录入页：说明数据仅存数学特征向量，不储存照片原图。
- 签到页：光线充足、摘掉口罩（若组织者政策允许）、正对摄像头。
- `GEO_FACE`：按钮文案强调「地理+人脸一次性提交」，避免误以为只点地理即完成。

---

## 7. 数据与枚举

### 7.1 `sessions.checkin_modes` 扩充

取值：`GEO` | `QR` | `BOTH` | `FACE` | `GEO_FACE`。

### 7.2 `checkin_records.method`

新增：`face` | `geo_face`（小写与其它记录一致）。

### 7.3 失败码

| code | 场景 |
|------|------|
| `face_not_enrolled` | 未录入样本 |
| `face_mismatch` | 向量与样本距离超出阈值 |
| `no_descriptor` | 请求未提供合法 descriptor |

## 8. 非目标（本版仍不包含）

- 活体检测（眨眼/摇头）、静默活体、静默硬件 trust。
- 与教务/一卡通/门禁系统对接。
- 大规模人脸库 1:N 搜索。

---

## 9. 上线与运营检查

- **HTTPS**：生产环境必须 HTTPS，否则摄像头/模型加载不可用（与既有地理签到提示一致）。
- **模型 CDN**：默认使用 jsDelivr 托管的 face-api 官方 weights；内网离线环境需自备静态资源路径（前端常量可改）。
- **隐私**：用户协议若有需要，增补「人脸识别特征数据处理说明」一节（法务审核）。

# 平台超级管理员 — 内容与账号治理 PRD

**文档版本**：1.0  
**状态**：已实现（与仓库接口一致）  
**关联**：`AI_DEV_PRODUCT_SPEC.md`（原「组织者 / 参与者」双角色）；本节定义 **单机部署可用的平台运维能力**，不涉及多租户计费与公安对接。

---

## 1. 背景与目标

### 1.1 问题

当出现 **违规组织命名、骚扰性活动名义、或通过活动/组织链路传播不当信息** 时，普通组织者 **无权**处置他人创建的组织与活动；若没有 **平台超级管理员**，只能人工改数据库，不可审计且易出错。

### 1.2 目标（MVP）

| 维度 | 内容 |
|------|------|
| **身份** | 由部署方在环境变量中 **显式配置**「超级管理员用户 id 列表」（不开放自助申请，避免社工与权限膨胀）。 |
| **组织能力** | 超级管理员可 **解散并删除任意组织**：清理成员关系、入会申请记录，并从所有活动中 **摘除该组织的名单引用**，避免残留的非法 `roster_org_ids`。 |
| **账号能力** | 可对用户 **封号**：禁止登录与一切需鉴权接口（旧 JWT 作废逻辑通过每次请求服务端重载用户信息实现封号拦截）。 |
| **内容载体（站内「消息」）** | 当前产品 **无独立聊天/信息流**；**UGC 主要落在「活动会话」**：活动标题 etc. Super admin 可对任意活动执行 **下架（取消 activity）**：`cancelled=1`，全员不可再继续签到发起侧展示为已取消（与组织者自行下线一致语义）。 |

### 1.3 非目标

- **不在 PRD 内**实现用户举报工单、语义审核模型、多级审核流。
- **不删除**已成功签到履历（考勤与纠纷取证）；下架活动 ≠ 清空 `checkin_records`。
- **不替换**法务流程：封号与解散组织应保留理由与审计日志备查。

## 2. 角色与安全模型

### 2.1 谁来当超级管理员

- 部署时在 `.env` 配置 **`SUPER_ADMIN_USER_IDS`**：逗号分隔的用户 **UUID**。仅这些账号在服务端被识别为 `is_super_admin`。
- 创建方式：先在站内 **正常注册**，将生成账号的 **`id`** 拷入 `.env`**，重启 API** 后生效。

### 2.2 封号对象保护

以下账号 **不允许被其他超级管理员封号**（防误操作系统账号）：

- `SUPER_ADMIN_USER_IDS` 内全部 id。

可自行解封或通过 DB 纠错（极少数场景）。

### 2.3 权限边界（禁止）

- Super admin **不**通过浏览器接口直接 **物理删除 SQLite 库文件**；
- 「删除用户账号行」暂不开放（封号已足够冻结行为；GDPR 式删除单列 Phase 2）。

## 3. 功能需求明细

### 3.1 解散组织（删除组织）

**前置**：组织存在。

**结果**：

1. 删除 `organization_join_requests`、`organization_members`、`organizations` 中该行。
2. 扫描 **`sessions.roster_org_ids`**（JSON 数组），从 **所有会话**中移除该 `org_id`，写回；
3. 写 **审计**：`delete_org`。

**不改变**：活动发起人、活动其它字段。

### 3.2 封号 / 解封

**封号**：`account_status=banned`，记录 `banned_at`、事由 `banned_reason`（必填）、操作人 `banned_by`（.actor super admin id）。

**解封**：恢复 `active`，写审计 `unban_user`。

登录失败提示：账号已停用（不泄露是否「被封」与其它用户差异细节 optional — MVP 明文提示即可）。

### 3.3 活动下架（处理「违规活动内容」）

**动作**：将该 session `cancelled=1`。  
**不改变**历史统计数据（已签记录保留）。  
可选 **下架说明**记入审计 payload。

### 3.4 审计日志

仅存 **操作者与目标、动作枚举、事由、UTC 时间**；不向普通用户暴露。

管理员控制台提供 **近期操作列表**（MVP：`GET /api/admin/audit-log?limit=`）。

## 4. 接口约定（概要）

| 方法与路径 | 说明 |
|-------------|------|
| `GET /api/users/me` | 扩展返回 `account_status`、`is_super_admin`。 |
| `DELETE /api/admin/organizations/:id` | Body: `{ reason }` |
| `POST /api/admin/users/:id/ban` | `{ reason }` |
| `POST /api/admin/users/:id/unban` | `{}` |
| `POST /api/admin/sessions/:id/cancel` | `{ reason? }` |
| `GET /api/admin/overview` | 用户数、组织数、活动数等 |
| `GET /api/admin/audit-log` | 最近审计 |

所有 `/api/admin/*`：`Authorization` + **`SUPER_ADMIN_USER_IDS` 命中**。

---

## 5. 验收标准

1. **非列表内用户**访问 `/api/admin/**` 一律 **403**。  
2. **被封用户**无法用密码登录，`authRequired` 请求一律 **403**（含旧 token）。  
3. **删除组织后**，任一 session 的 `roster_org_ids` 不再含该 id。  
4. **下架活动后**，参与者端不可签到（与原 `cancelled` 行为一致）。  
5. **审计表**每笔治理操作可查。

---

## 6. 与「消息」的口径

暂无独立站内信；本 PRD 所称 **处置违规内容与消息**，在 MVP 中等价为：

- **组织名称 / 简介类**：通过 **解散组织** 清除组织实体；
- **活动标题与传播路径**：通过 **下架活动** 停用展示与签到；

若日后增加评论/弹幕，应在同一 **审计与封号**模型上挂钩，复用本平台治理策略。

---

## 7. 控制台信息与交互（UI/UX）

### 7.1 业务范围分区（避免单页信息过载）

按 **五条线**分标签展示，一次只做一类事：

| 分区 | 目的 | 主要能力 |
|------|------|----------|
| **概览** | 看平台量级与风险提示 | 只读概要数字；简述治理边界 |
| **组织** | 处置组织实体与命名 | **搜索名称或完整 UUID** → 解散并删除 |
| **用户** | 处置帐号 | **帐号状态筛选（全部/正常/已停用）** + **关键词（≥2 字）** → 封号 / 解除停用 |
| **活动** | 处置 UGC 活动名义 | **状态（全部/仅有效/仅已下架）** + **本地筛选标题或 id** → 下架（取消），展示发起人 |
| **审计** | 追责与回溯 | **动作类型筛选**；列表含时间、动作、对象 id、执行人用户名、事由；可复制 id |

### 7.2 检索心智模型

- **列表级筛选**（服务端）与 **单行内关键字**（活动表为前端即时过滤）分离，减轻「我到底在搜哪一种」的负担。
- 审计日志 **最近优先**，支持按 **`action`** 枚举缩小排查面。

### 7.3 高危操作与共通反馈

解散组织、封号、下架活动均需 **二次确认**；成功后 **刷新当前分区** 并 **同步刷新审计摘要**。

### 7.4 视觉一致性

控制台页使用与全站一致的 **导航壳、分段控件（tabs）、分组列表与安全色**，不另起设计语言。

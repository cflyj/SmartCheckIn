# 人脸比对误识别抑制 — 产品与技术说明（PRD）

**文档版本**：1.0  
**关联实现**：`server/services/faceVerify.js`、`src/utils/faceClient.js`、`POST /sessions/:id/checkin/face`、`POST /sessions/:id/checkin/geo-face`、`POST /users/me/face`  
**与既有文档**：`FACE_CHECKIN_PRD.md`（识别形态与阈值原则）、`FACE_SIGNIN_ANTI_PROXY_PRD.md`（录入治理与临场换样本）互为补充。**本文聚焦于「同人 / 非同人误判」的工程收敛，而非活体或公安级身份。**

---

## 1. 问题陈述

### 1.1 现象

在连续两次签到测试中，**使用两张不同自然人的人脸图片均可能「比对通过」**。用户侧观感为：**系统无法区分换人代签**。  

### 1.2 根本原因（诚实边界）

当前实现基于浏览器 **face-api.js** 输出的 **128 维特征向量**，服务端做 **欧氏距离（L2）与阈值** 判别：

- **无硬件 / 服务端活体**：无法保证「取景框内是活体本人」；翻拍、视频等攻击不在本篇 MVP 解决范围内（见§4 非目标）。
- **阈值过松**：在 faceRecognitionNet 的度量下，陌生人之间的 L2 距离**仍存在一定概率**落入「过宽」阈值，呈现为**错误接受（False Accept）**。
- **检测器召回优先**：TinyFaceDetector 若以较低 `scoreThreshold` 运行，易产生**含糊检测框**，其后续 descriptor 判别力下降，进一步放大误判。

本产品**不宣称**能将「非同人误判」归零；目标是在现有架构下 **可配置地收紧误判率**，并以文档与产品有义务披露能力与局限。

## 2. 目标与非目标

| 维度 | 内容 |
|------|------|
| **目标（MVP）** | ① **默认收紧** L2 比对阈值（在误拒可接受前提下降低误纳）；② **提高人脸检测阈值** 与 **前置拒绝低置信度帧**（不传低质量 descriptor）；③ 服务端可对 **探测器置信度** 做阈值校验（可配合「强制附带置信度」部署开关）；④ 环境与前端可配置的 **显性参数**，便于运维按场景微调。 |
| **非目标** | 活体检测 SDK、双摄深度、静默活体、与公安人脸库比对、组织级活体策略引擎；本节不替代 `FACE_SIGNIN_ANTI_PROXY` 中对「换人改绑样本」的频率治理。 |

## 3. 产品与体验要求

### 3.1 阈值与文案

- **比对失败**：继续返回 `422` + `face_mismatch`（或等价业务码），用户文案保持「与已录入人脸不一致…」一类表述。
- **检测置信度过低**：返回 `422` + `low_detection_confidence`，提示用户对光、正对、保持稳定。
- **运维披露**：在活动说明或站内帮助中可读形式说明：**强光线、正脸**有助于提高通过率；逆光/戴口罩可能导致 **误拒** 增加（不可简单把阈值无限收紧）。

### 3.2 兼容性

- 旧版前端**不传** `detection_score` 时：默认仍可走 **仅向量比对**；部署方可通过环境变量 **要求附带置信度**（见§5），用于强制升级版客户端后再开放。

---

## 4. 技术方案概要

### 4.1 服务端：`FACE_MATCH_DISTANCE_THRESHOLD`

- **含义**：L2 **小于等于**该值则视为同人（与原实现一致）。
- **默认**：比历史 `0.55` **更严格**（实现以代码常量为准）。
- **范围**：在安全范围内裁剪（过小则普遍误拒，过大则误纳上升）；具体上下限见源码注释。

### 4.2 服务端：`FACE_MIN_PROBE_DETECTION_SCORE` / `FACE_REQUIRE_DETECTION_SCORE`

- **含义**：客户端上报的 **`detection_score`**（通常为 TinyFace detector 的输出分数，约在 0～1）。
- **`FACE_MIN_PROBE_DETECTION_SCORE`**：当请求体中包含 `detection_score` 时，**低于则拒绝**比对。
- **`FACE_REQUIRE_DETECTION_SCORE`**：为 true 时，**缺少**合法 `detection_score` 直接拒绝（用于全站升级到新前端后）。

### 4.3 前端：`faceClient.captureFaceDescriptor`

- 提高 **TinyFaceDetector** 的 `scoreThreshold`，减少含糊框。
- 在生成 descriptor 时读取 **`detection.score`**，低于 **客户端下限**（可 `VITE_*` 配置）则 **不写库、不提交**，直接提示用户重采。
- **安全说明**：置信度可被篡改，不应作为唯一防线；主线仍是 **服务端 L2 + 服务端阈值**；置信度字段用于一致性与质量门禁。

## 5. 配置项一览

### 服务端（Node / `.env`）

| 变量 | 含义 | 备注 |
|------|------|------|
| `FACE_MATCH_DISTANCE_THRESHOLD` | L2 判定同人上限 | 未设置则用默认更严常量 |
| `FACE_MIN_PROBE_DETECTION_SCORE` | `detection_score` 服务端下限 | 未设置则用默认 |
| `FACE_REQUIRE_DETECTION_SCORE` | 是否强制请求带 `detection_score` | `1` / `true` 等 |

### 前端（Vite，`import.meta.env`）

| 变量 | 含义 |
|------|------|
| `VITE_FACE_DETECTOR_SCORE_THRESHOLD` | TinyFace 检测阈值（越高越保守） |
| `VITE_FACE_MIN_DETECTION_SCORE` | 客户端拒绝提交的最小置信度 |

---

## 6. 验收标准

1. **默认配置**下，使用明显不同二人的连续采样：**错误接受概率应较旧版阈值显著降低**（回归以人工抽测为主，不写死自动化「两人必不过」）。
2. **同人、光线正常**：仍能稳定完成签到与录入。
3. `FACE_MATCH_DISTANCE_THRESHOLD` 调整可在**不改前端**前提下改变误纳 / 误拒权衡。
4. 前端在 **逆光 / 极侧脸** 时更多出现「未检测到 / 置信度过低」提示，而非错误通过。

## 摘要

传统课堂与活动签到依赖纸质点名或人工统计，效率低且难以追溯。本文围绕毕业设计题目，设计并实现一套基于浏览器的智能签到系统 **SmartCheckIn**：组织者可在 Web 端创建签到活动、配置地理围栏与动态二维码策略；参与者使用移动端浏览器完成地理位置签到与/或扫码签到；系统记录每次尝试的结果与原因，并支持按活动查看统计与明细。系统采用 **Vue 3 + Vite** 构建单页应用，**Node.js + Express** 提供 RESTful 接口，**SQLite** 持久化数据，**JWT** 完成身份认证。中期阶段已完成核心功能开发与服务器端一体化部署验证，后续将补充测试用例、完善论文正文与答辩材料。

**关键词**：智能签到；地理位置服务；二维码；Vue；Express；SQLite

## 第一章 绪论

### 1.1 选题背景

高校教学与各类校园活动中，签到是考勤与过程管理的基础环节。常见方式包括课堂点名、纸质签到表、微信群接龙等，存在耗时、代签难以防范、数据不易汇总等问题。随着智能手机普及，基于 **HTML5 地理定位** 与 **二维码** 的签到方案可在不安装专用 App 的前提下完成身份与时间、空间维度的校验，具有部署轻、迭代快的特点。

### 1.2 研究意义与应用价值

本设计的意义主要体现在：（1）将签到流程线上化，减轻组织者负担；（2）通过地理围栏与动态令牌等机制，在 Web 技术约束下提供基础的防作弊能力；（3）签到记录结构化存储，便于统计与事后审计。应用上适用于小班课程、讲座、社团活动等对实时性要求较高的场景。

### 1.3 主要工作内容

本人主要完成：需求分析与功能边界确定；前后端总体架构与数据库设计；用户注册登录与 JWT 鉴权；活动（会话）管理；地理签到与二维码签到业务流程；组织与邀请相关能力；参与者与组织者两端界面实现；开发环境联调与生产环境部署文档整理。中期报告撰写时，核心功能已可端到端演示。

### 1.4 论文结构安排

第二章介绍需求分析；第三章介绍总体设计；第四章汇报中期已完成工作与成果；第五章说明关键技术实现与遇到的问题；第六章给出后期工作计划。全文以工程实现为主线，与仓库内产品规格、部署说明及工程复盘文档相互印证。

## 第二章 需求分析

### 2.1 用户角色

系统涉及两类主要角色：

- **组织者**：创建与管理签到活动，配置签到方式（纯地理、纯二维码或二者兼有），查看大屏二维码、签到统计与记录明细，复制参与者入口链接等。
- **参与者**：在活动时间窗内查看可参与活动，完成地理定位和/或扫码（含拍照识别、手动输入令牌等降级方式）签到，查看个人签到结果。

此外，系统支持 **组织（Organization）** 实体：用户可创建组织、通过加入码加入组织，活动可关联名单制与组织成员，以满足「仅允许特定人群签到」的需求。

### 2.2 功能需求

| 编号 | 功能模块       | 描述 |
|------|----------------|------|
| F1   | 账户与认证     | 用户注册、登录；密码哈希存储；登录后颁发 JWT，前端请求携带 `Authorization: Bearer` |
| F2   | 活动管理       | 创建/编辑活动，设置开始与结束时间、签到方式、地理围栏参数、二维码策略、参与者范围与邀请码等 |
| F3   | 地理签到       | 浏览器获取经纬度与精度；服务端根据 Haversine 公式判断是否在允许半径内；时间窗外拒绝 |
| F4   | 二维码签到     | 组织者端展示含动态令牌的签到链接二维码；参与者扫码或解析 URL 中 `token` 提交；服务端校验有效期与重放策略 |
| F5   | 签到记录与统计 | 记录成功/失败及失败原因码；组织者查看统计与记录列表 |
| F6   | 组织与成员     | 组织创建、加入、成员管理；与活动名单制联动（以实际实现为准） |

### 2.3 非功能需求

- **移动端优先**：界面适配手机浏览器，主要流程在移动端完成。
- **安全上下文**：地理定位、摄像头等能力依赖 HTTPS 或 localhost；需在文档与用户提示中明确。
- **兼容性**：部分国产浏览器对剪贴板 API、摄像头支持不完整，需提供复制降级与替代扫码方式。
- **可部署性**：支持在 Linux 服务器上使用 Node 进程与 Nginx 反向代理部署；生产环境由 Express 托管前端静态资源与 API，避免 CORS 与路径混乱。

### 2.4 可行性分析

技术上所选栈均为成熟开源方案，开发文档齐全；SQLite 适合毕业设计规模的数据量与单机部署。经济上无需商业地图 SDK 即可完成基于经纬度的距离判定（若后续接入高德等可再扩展）。操作上组织者仅需浏览器与大屏或投影即可展示二维码。

## 第三章 总体设计

### 3.1 系统架构

系统采用 **B/S 架构**。用户通过浏览器访问前端单页应用，前端通过 HTTP 调用后端 `/api` 下 JSON 接口；后端读写 SQLite 数据库并执行业务规则。

逻辑分层如下：

- **表现层**：Vue 3 组件 + Vue Router 路由 + Pinia 状态管理。
- **应用层**：Express 路由模块（认证、用户、组织、会话与签到等）。
- **数据层**：`node:sqlite` 同步数据库接口，库文件默认位于 `server/app.db`。

开发环境下，Vite 开发服务器将 `/api` **代理**至本机 Node 端口，避免跨域问题。生产环境下，前端构建产物置于 `dist/`，由 Express 在 `NODE_ENV=production` 时提供静态文件，并对非 API 路径回退至 `index.html`，以支持前端路由。

### 3.2 功能模块结构

- **认证模块**：注册、登录、JWT 签发与校验中间件。
- **会话模块**：活动的增删改查、状态与时间窗判断、地理与二维码配置序列化存储。
- **签到模块**：地理校验、二维码令牌生成与校验、防重复成功签到（数据库唯一索引约束等）。
- **组织模块**：组织、成员、加入策略等（与 `server/db.js` 中表结构一致）。
- **前端模块**：按路由划分为首页、参与者活动列表与详情、组织者控制台、活动表单、二维码展示页、统计与记录页、组织相关页面等。

### 3.3 数据库概念设计

核心实体包括：**用户（users）**、**签到会话（sessions）**、**签到记录（checkin_records）**、**会话二维码状态（session_qr_state）**、**组织（organizations）** 及成员、加入请求等关联表。会话与记录一对多；同一用户对同一会话的成功签到在逻辑上唯一，并通过部分唯一索引等技术手段保证数据一致性。

### 3.4 技术选型说明

| 层次   | 技术选型              | 选型理由 |
|--------|-----------------------|----------|
| 前端框架 | Vue 3                 | 组合式 API、生态成熟、适合快速构建交互界面 |
| 构建工具 | Vite                  | 开发热更新快，与 Vue 官方推荐一致 |
| 路由与状态 | Vue Router、Pinia   | 官方推荐，结构清晰 |
| 扫码能力 | html5-qrcode          | 支持摄像头与文件识别，满足参与者端需求 |
| 二维码生成 | qrcode（服务端/前端按实现） | 生成大屏展示用二维码图片或数据 URL |
| 后端运行时 | Node.js LTS         | 与前端同语言，便于全栈开发与部署 |
| Web 框架 | Express               | 中间件模型简单，适合 REST API |
| 数据库   | SQLite（node:sqlite） | 零配置、单文件、易备份，适合课题与小型部署 |
| 安全     | bcryptjs、jsonwebtoken | 密码哈希与无状态认证的行业常见组合 |

## 第五章 关键技术实现与问题分析

### 5.1 前后端分离与开发代理

开发阶段前端运行于 Vite 默认端口，若浏览器直接请求后端端口会触发 **CORS** 限制。本项目在 `vite.config.js` 中配置 `server.proxy`，将 `/api` 转发至 `127.0.0.1:API_PORT`，与后端监听端口保持一致。该方式不改变前端代码中的相对路径写法，降低环境切换成本。

### 5.2 生产环境 SPA 路由与静态资源

采用 `createWebHistory()` 时，用户刷新深层路径（如 `/participant/sessions/xxx`）会向服务器请求该路径。若服务器未配置回退，将返回 404。本项目在生产环境由 Express 对非 `/api` 请求返回 `index.html`，由前端路由接管，从而解决该问题。

### 5.3 地理签到与用户体验

地理定位在室内可能存在精度差、首次定位慢等问题。实现上采用 `watchPosition` 持续获取位置，并结合**最长等待时间**与**精度改善停滞**等策略，在准确性与等待时间之间折中。用户拒绝定位权限时，返回明确错误码与界面提示，引导用户到系统设置中授权。

### 5.4 二维码载荷与扫码兼容性

大屏二维码内容为**完整签到 URL**，不同扫码软件返回结果可能为整段字符串而非纯 token。项目在 `src/utils/qrPayload.js` 与 `server/utils/qrPayload.js` 中保持一致的解析逻辑：优先从 URL 的 `token` 查询参数提取，失败时再回退到正则等方式，并在服务端再次校验，减少客户端差异导致的失败。

### 5.5 剪贴板 API 在部分浏览器中的失败

`navigator.clipboard.writeText` 在非安全上下文或部分壳浏览器中不可用。项目抽取 `copyToClipboard` 工具函数：优先使用 Clipboard API，失败则使用 `textarea` 与 `document.execCommand('copy')` 降级；仍失败时在界面展示只读输入框，提示用户长按全选复制或更换浏览器。

### 5.6 HTTPS 与移动端敏感能力

摄像头、定位等能力在真实移动场景中常要求 **HTTPS**。文档中说明了使用正式域名证书部署，以及使用 Cloudflare Quick Tunnel 进行临时 HTTPS 验证的思路，保证中期演示可在真机上完成地理与扫码流程。
